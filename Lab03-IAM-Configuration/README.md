# Lab 03 – IAM Configuration and Security Hardening

## Objective
Replace root account usage with a properly structured IAM 
configuration following least privilege principles. Implement 
security controls including password policy, MFA enforcement, 
and access monitoring.

## Services Used
- AWS IAM
- IAM Access Analyzer
- IAM Credential Report

## What I Built

### User Structure
| User | Group | Policy | Access Level |
|------|-------|--------|--------------|
| admin_sean | Administrators | AdministratorAccess | Full admin |
| dev-user01 | Developers | PowerUserAccess | No IAM access |
| auditor-user01 | Auditors | ReadOnlyAccess | Read only |

### Password Policy
| Setting | Value | Reason |
|---------|-------|--------|
| Minimum length | 12 | Industry standard |
| Complexity | All four types | Required variety |
| Expiration | Off | Modern NIST guidance |
| User self-service | On | Reduces admin overhead |
| Prevent reuse | 5 | Stops password cycling |

### MFA Enforcement Policy
- Created custom policy: RequireMFA
- Attached to Developers and Auditors groups
- Uses Deny + NotAction pattern to block all actions 
  until MFA is configured
- Explicit Deny always overrides Allow in IAM evaluation

### Security Monitoring
- IAM Credential Report: identifies users without MFA, 
  unused passwords, and unrotated access keys
- IAM Access Analyzer: scans for resources accessible 
  outside the account

## Key Concepts Learned
- Least privilege — users get only what their job requires
- Groups are not identities — cannot log in as a group
- Explicit Deny always wins over Allow in policy evaluation
- Privilege escalation prevention — developers cannot 
  modify IAM to elevate their own permissions
- PowerUserAccess blocks IAM and Organizations actions
- Credential Report is the correct tool for auditing 
  MFA and password compliance at scale
- Autogenerated temp passwords + force reset = correct 
  provisioning pattern for new users

## Exam Relevance
- Domain 4: Security and Compliance (16%)
- IAM policy evaluation order: Explicit Deny > Allow > Implicit Deny
- Credential Report for bulk MFA/password auditing
- Access Analyzer for detecting external resource exposure
- MFA enforcement via Deny policy is a production pattern
- Root account: remove keys, enable MFA, never use daily
